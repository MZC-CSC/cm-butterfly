# 02. 실행

---

## 한 줄 요약

```bash
cd front
npx bddgen --config tests/e2e/playwright.config.ts        # .feature → 테스트 생성
npx playwright test --config=tests/e2e/playwright.config.ts
```

기본 실행은 **seed + functional** 이다. 실제 클라우드 자원을 만드는 것(`@costly`)은 **빠져 있다.**

---

## 3단 구성 — 무엇이 언제 도는가

```
seed ─┬─→ functional      (Playwright project dependencies 로 순서를 명시)
      └─→ scenario
```

| 프로젝트 | 무엇 | 기본 실행 | 걸리는 시간 |
|----------|------|:--------:|------------|
| **seed** | 기능 테스트가 딛고 설 데이터를 실제로 만든다 — 소스 등록 → 수집 → 소스모델(인프라·SW) → 요금 안전 워크플로우(DAG 등록까지 확인) | ✅ | ~40초 |
| **functional** | 화면별 기능 검증(`@unit`) | ✅ | ~2분 |
| **scenario** | 마이그레이션 전체 흐름. **실제 EC2를 만든다**(`@costly`) | ❌ (옵트인) | ~10분 |

### 왜 seed가 필요한가

`@unit` 은 **mock 없이 실제 연계 프레임워크를 친다**(연계 계약이 깨지는 걸 잡는 게 목적이다).
그래서 소스 모델·워크플로우가 *실제로 있어야* 한다. 예전엔 그 데이터를 만들어 주는 주체가 없어서
`e2e-nano-source` 같은 이름을 기다리다 타임아웃으로 죽었다. seed가 그걸 만든다.

---

## 실행 예

```bash
cd front

export BASE_URL=http://cmig.dev.cscmzc.com
export TEST_SOURCE_IP=<소스 VM 공인 IP>
export TEST_SOURCE_SSH_USER=ubuntu
export TEST_SOURCE_PRIVATE_KEY="$(cat e2e-source-key)"

npx bddgen --config tests/e2e/playwright.config.ts

# 1) 기본 — seed + functional (클라우드 자원 안 만듦)
npx playwright test --config=tests/e2e/playwright.config.ts

# 2) 전체 시나리오 — 실제 EC2 생성 (요금 발생)
E2E_INCLUDE_COSTLY=1 npx playwright test --config=tests/e2e/playwright.config.ts --project=scenario

# 3) 일부만
npx playwright test --config=tests/e2e/playwright.config.ts --grep @smoke
npx playwright test --config=tests/e2e/playwright.config.ts --grep @workflow
npx playwright test --config=tests/e2e/playwright.config.ts --grep "@source"
```

> `--grep` 을 써도 **seed 는 항상 먼저 돈다**(project dependency). 데이터 없이 도는 일은 없다.

---

## 태그

| 태그 | 뜻 |
|------|-----|
| `@seed` | 데이터 준비 |
| `@unit` | 기능 테스트. **mock 없이 실 라인업** |
| `@scenario` `@e2e` | 전체 흐름 |
| `@costly` | **실제 클라우드 자원을 만들거나 지운다.** 기본 제외, `E2E_INCLUDE_COSTLY=1` 일 때만 |
| `@smoke` | 릴리스 전 최소 확인 |
| 도메인 | `@auth @source @models @workflow @workload @cloudResources @migration @software` |

## 요금 관련 주의

- 시나리오는 **실제 EC2를 만든다.** 추천은 저비용(nano/small급)으로 강제하지만 공짜는 아니다.
- 시나리오 끝의 정리(teardown)가 인프라·소스그룹을 지운다. **중간에 실패하면 정리가 안 돌 수 있다** —
  실패했을 때는 워크로드 화면에서 남은 인프라를 직접 지운다.
- 소스 VM은 테스트가 만들지 않는다. 사람이 만들고, 다 쓰면 사람이 내린다.

# 05. 증거와 리포트

---

## 왜 성공한 테스트도 화면을 남기나

**"PASS" 한 줄은 증거가 아니다.**

단언문이 통과해도 화면이 정상이라는 뜻은 아니다. 이 프로젝트에서 실제로 이런 일이 있었다.

| 겉보기 | 실제 |
|--------|------|
| `타깃 EC2 인스턴스가 정상 생성된다` PASS | 인프라 행은 있었지만 **노드가 Terminated** 였다 — 이번 실행은 EC2를 하나도 만들지 못했다 |
| `소스그룹 목록에 ... 이 보인다` PASS | **이전 런이 남긴 그룹**에 정규식이 우연히 걸린 것이었다 |
| `워크플로우 실행 이력이 생성된다` PASS | *워크플로우 목록* 표를 보고 있었다. 그 표는 실행 0건이어도 늘 떠 있다 |

캡처된 화면 한 장이 이걸 드러냈다:

```
row "infra101 infra101 Terminated:1 (R:0/1) aws 1 0 0 1"
```

그래서 **실패 건만이 아니라 모든 테스트**의 화면·영상을 남긴다. 사람이 눈으로 판정할 수 있어야 한다.

## 무엇이 남는가

| 산출물 | 설명 |
|--------|------|
| `01-before.png` | **시작 화면** — 테스트 전에 무엇이 보였나 |
| `99-after.png` | **종료 화면** — 끝났을 때 화면 상태 |
| `video.webm` | 테스트 전 과정 영상 — 어느 단계에서 무엇이 보였는지 그대로 |
| `trace.zip` | 액션별 DOM·네트워크·콘솔. 되짚어볼 수 있다 |
| 단계별 캡처 | 시나리오만 — `10-소스모델-저장됨`, `20-커스텀모델-생성됨`, `30-타깃모델-저장됨`, `40-워크로드-생성됨`, `50-인프라-상세`, `60-노드-목록`, `70-부하테스트-결과` |

저장 위치: `front/test-results/` · 모아 보기: `front/playwright-report/`

```bash
npx playwright show-report front/playwright-report
```

## 끄기

용량이 부담되는 CI 등에서는 실패 건만 남기도록 줄인다.

```bash
E2E_EVIDENCE=0 npx playwright test --config=tests/e2e/playwright.config.ts
```

| | `E2E_EVIDENCE` 기본(켜짐) | `E2E_EVIDENCE=0` |
|---|---|---|
| screenshot | `on` (모든 테스트) | `only-on-failure` |
| video | `on` | `retain-on-failure` |
| trace | `on` | `retain-on-failure` |

---

## 시나리오에 캡처 지점 추가하기

`.feature` 에 한 줄 넣으면 된다.

```gherkin
그리고 화면을 "40-워크로드-생성됨" 이름으로 캡처한다
```

번호를 앞에 붙이면 리포트에서 순서대로 정렬돼 **전/후 비교**가 쉽다.

---

## 요청 기록으로 계약 검증하기

화면이 멀쩡해 보여도 **나가는 요청**이 구 스키마면 실제로는 깨진 것이다.
`support/fixtures.ts` 가 콘솔이 프록시로 보낸 요청을 모아 두고, 스텝에서 검증할 수 있다.

```gherkin
그리고 콘솔이 구 스키마 요청을 보내지 않는다
```

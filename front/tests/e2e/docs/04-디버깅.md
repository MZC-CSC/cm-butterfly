# 04. 디버깅

실패했을 때 **무엇부터 보는지** 순서대로.

---

## 0. 먼저 — "PASS 라고 다 된 게 아니다"

이 프로젝트가 실제로 겪은 일이다.

- `타깃 EC2 인스턴스가 정상 생성된다` 가 **노드가 죽은(Terminated) 인프라를 보고 통과**하고 있었다.
  행이 보이기만 하면 통과였고, cm-beetle 이 타깃 이름을 늘 `infra101` 로 만드는 탓에 *앞선 실행이 남긴 행*이
  계속 그 단언을 만족시켰다.
- 소스그룹 확인이 **이전 런이 남긴 그룹**에 정규식으로 우연히 매칭돼 통과했다.
- 실행 이력 확인이 *워크플로우 목록* 테이블을 보고 있었다. 그 표는 실행이 0건이어도 늘 떠 있어 **언제나 통과**했다.

그래서 **성공한 테스트도 화면·영상을 남긴다.** 초록불을 봤다고 끝내지 말고, 의심스러우면 리포트에서 화면을 열어 본다.
→ [05-증거와리포트](05-증거와리포트.md)

## 1. HTML 리포트

```bash
npx playwright show-report front/playwright-report
```

테스트별로 **시작 화면(`01-before`) · 종료 화면(`99-after`) · 영상 · trace** 가 붙어 있다.
시나리오는 단계별 캡처(`10-소스모델-저장됨`, `40-워크로드-생성됨`, `70-부하테스트-결과` …)도 있다.

## 2. trace 뷰어 — 가장 강력하다

```bash
npx playwright show-trace front/test-results/<테스트폴더>/trace.zip
```

액션별로 **그 순간의 DOM·네트워크 요청·콘솔**을 되짚을 수 있다. 셀렉터가 왜 못 잡았는지 여기서 대부분 끝난다.

## 3. 눈으로 보며 돌리기

```bash
npx playwright test --config=tests/e2e/playwright.config.ts --grep @source --headed
npx playwright test --config=tests/e2e/playwright.config.ts --ui        # UI 모드(추천)
npx playwright test --config=tests/e2e/playwright.config.ts --debug     # 스텝별 정지
```

## 자주 나는 실패와 원인

### "정의되지 않은 스텝"
`npx bddgen` 에서 걸린다. `.feature` 문장과 `steps/*.ts` 의 문구가 한 글자라도 다르면 매칭되지 않는다.

### 목록에 방금 만든 게 없다고 한다
목록은 **15행씩 끊긴다.** 게다가 상단 검색창은 mirinae query 태그 방식이라 **이 테이블들에는 필터가 붙어 있지 않다**
(입력해도 목록이 안 줄어든다). 그래서 새로 만든 행이 2페이지로 밀린다.

→ `support/pagination.ts` 의 `TablePagination` 이 **페이지를 넘겨 가며** 찾고,
   **몇 페이지에서 찾았는지 로그로 남긴다**: `[pagination] "e2e-src-group-647993" → 3/3 페이지에서 확인`

새 목록 화면을 다룰 때는 이 헬퍼를 쓴다. 1페이지만 보고 "없다"고 단정하면 안 된다.

### 등록은 됐는데 목록에 안 보인다
등록 직후 목록이 **자동 갱신되지 않는** 화면이 있다. Page Object 가 조회 전에 새로고침한다
(`SourceServicesPage.revealGroup`). 새 화면에서 같은 증상이면 같은 처리를 넣는다.

### strict mode violation (요소가 2개 이상 잡힘)
`.or(getByPlaceholder(...))` 같은 **문구 기반 fallback** 때문인 경우가 많다.
실제로 `getByPlaceholder('Time')` 이 "Time" 과 "Test Run Time" 두 입력칸에 모두 걸려 죽었다.

→ **testid만 쓴다.** testid가 없으면 **화면 소스에 부여한다.** 문구에 기대지 않는다.

### PButtonModal 에 testid를 달았는데 안 잡힌다
`data-testid` 를 `p-button-modal` 에 달면 **래퍼**에 붙어서 Playwright 의 "보이는 요소"로 잡히지 않는다.
(`load-config-modal`·`mci-delete-modal`·모델 저장 폼에서 전부 같은 함정을 밟았다.)

→ 모달의 열림/닫힘은 **안에 있는 실제 요소**(이름 입력칸 등)로 판정한다.

### 워크플로우 실행이 `provided dag_id is not exist` 로 거부된다
cm-cicada 는 워크플로우를 만들 때 DAG YAML 을 디스크에 쓰고, **airflow 가 그걸 주기적으로 파싱해 등록**한다.
등록 전(수십 초)에 실행하면 거부된다.

→ seed 가 요금 안전 워크플로우를 만든 뒤 **실행이 받아들여질 때까지 확인**하고 넘어간다.
   마이그레이션 워크플로우는 **재시도하지 않는다**(누를 때마다 EC2가 늘어난다). 대신 충분히 기다린 뒤 한 번만 실행한다.

### airflow 가 DAG 를 하나도 등록하지 않는다
gusty 의 `dags.py` 가 **DAG 디렉토리 전부를 한 모듈로 빌드**한다. 그래서 **깨진 DAG 하나가 전부를 막는다.**

```bash
docker exec airflow-server airflow dags list-import-errors
docker exec airflow-server airflow dags list
```

옛 버전이 만든 DAG 가 이미 없어진 오퍼레이터(`SimpleHttpOperator`)를 참조하면 전부 등록 실패한다.
그 DAG 디렉토리를 치우면 복구된다(생성 시 1회만 쓰이고 재생성되지 않는다).

### operationId 호출이 404 인데 화면은 그냥 "비어 보인다"
콘솔은 `POST /api/{operationId}` 로만 나가고 백엔드가 api.yaml 로 프록시한다.
**접두어(`{서비스}/`)가 없으면 404** 인데, 화면에서는 "목록이 비었다"처럼 보여 눈으로는 안 잡힌다.

```bash
npm run check:opids
```

콘솔·e2e 가 부르는 operationId 를 api.yaml 과 정적 대조해 **없는 이름·접두어 누락**을 잡는다.

### 원격명령(nginx 설치)이 대상을 못 찾는다
- 네임스페이스는 **`mig01`** 이다(`default` 아님). 콘솔의 `DEFAULT_NAMESPACE` 와 같아야 한다.
- 노드 SSH 계정은 **`cb-user`** 다(소스 VM 의 `ubuntu` 가 아니다). `GetInfra` 의 `nodeUserName` 을 그대로 쓴다.

# 01. 설치

e2e를 처음 돌리기 전에 한 번만 하면 되는 준비다.

---

## 1. 의존성 설치

```bash
cd front
npm install --legacy-peer-deps          # playwright-bdd · @playwright/test 포함
npx playwright install --with-deps chromium
```

`--legacy-peer-deps` 는 이 저장소의 기존 규칙이다(mirinae 계열 peer 충돌 회피).

---

## 2. 대상 서버 (BASE_URL)

테스트는 **띄워 놓은 콘솔**을 브라우저로 조작한다. 그 콘솔이 어디 떠 있는지 `BASE_URL` 로 알려준다.

| 대상 | BASE_URL | 언제 쓰나 |
|------|----------|-----------|
| 원격 개발 서버 | `http://cmig.dev.cscmzc.com` | **기본.** 연계 프레임워크가 다 떠 있어야 하므로 여기서 돌린다 |
| 로컬 docker | `http://localhost` | 로컬에 라인업을 다 띄운 경우 |
| 로컬 vite | `http://localhost:5173` | 화면만 빠르게 볼 때 (백엔드는 별도) |

> ⚠️ `@unit`·`@scenario` 는 **mock을 쓰지 않고 실제 연계 프레임워크를 친다.**
> 그래서 cm-honeybee·cm-beetle·cm-cicada·cm-damselfly·cm-ant·cb-tumblebug 이 함께 떠 있어야 한다.
> 프론트만 띄워 놓고 돌리면 대부분 실패한다.

## 3. ★ 소스 서버 VM (필수)

가장 자주 빠뜨리는 준비물이다.

시나리오는 "온프렘 서버를 클라우드로 옮긴다"를 검증한다. 그 **온프렘 서버 역할을 할 실제 VM**이 있어야 한다.
cm-honeybee 가 그 서버에 **SSH로 직접 붙어** 인프라·소프트웨어를 수집하기 때문에, 더미 IP를 넣으면 수집이 끝내
실패하고 그 뒤의 소스모델·추천·워크플로우가 전부 데이터 없이 무너진다.

### 요구사항

| 항목 | 값 | 이유 |
|------|-----|------|
| 크기 | **nano / micro / small** | 추천 스펙이 소스 스펙을 따라간다. 크게 만들면 큰 인스턴스를 추천받아 요금이 커진다 |
| OS | Ubuntu 22.04 | 수집·원격명령 검증 기준 |
| 접근 | 콘솔(cm-honeybee)이 있는 곳에서 **22 포트 SSH 가능** | 수집이 SSH로 이뤄진다 |
| 인증 | **전용 SSH 키** | 아래 주의 참조 |

### 주의 — 키는 이 VM 전용으로 만든다

honeybee는 넘겨준 개인키를 **자기 DB에 저장**한다. 그러니 개발 서버까지 열리는 공용 키를 주면 안 된다.
이 VM에만 통하는 키를 새로 만들어 넣는다.

```bash
# 전용 키 생성
ssh-keygen -t ed25519 -N "" -C "e2e-source-vm" -f e2e-source-key

# 소스 VM에 공개키 등록 (최초 접속은 인스턴스 생성 시 지정한 키로)
ssh -i <최초키>.pem ubuntu@<소스VM IP> \
  "echo '$(cat e2e-source-key.pub)' >> ~/.ssh/authorized_keys"

# 확인
ssh -i e2e-source-key ubuntu@<소스VM IP> 'echo OK'
```

이 개인키를 `TEST_SOURCE_PRIVATE_KEY` 로 넘긴다([03-환경설정](03-환경설정.md)).

> VM을 중지했다 다시 켜면 **공인 IP가 바뀐다.** `TEST_SOURCE_IP` 를 다시 잡아 준다.


## A. E2E 테스트 3분류 — 더미는 `@dummy` 로만

E2E 테스트는 성격에 따라 **세 종류**로 나눈다. 이 구분이 태그·프로젝트 구성에 그대로 드러나야 한다.

| 분류 | 태그 | 실서버 | 더미(mock·fixture 기본값) |
|------|------|:------:|:------------------------:|
| **더미(dummy)** | **`@dummy` 명시 필수** | 불필요 | **허용 — `@dummy` 일 때만** |
| **기능(function)** | `@unit` 등 | **필수** | **금지** |
| **시나리오(scenario)** | `@scenario`(+`@costly`) | **필수** | **금지** |

- **더미** — 실서버 없이 도는 검증(순수 UI 로직·셀렉터·라우팅·빈 상태 렌더링 등). 실 연계 프레임워크를 치지 않는다. **반드시 `@dummy` 로 그 사실을 드러낸다.**
- **기능** — 화면별 기능 검증. mock 없이 실제 연계 프레임워크를 쳐서 계약 파손을 잡는다.
- **시나리오** — 마이그레이션 전체 흐름. 실제 자원까지 만들어 end-to-end 로 검증한다.

### 규율

1. **`@dummy` 로 명시한 것만 더미를 쓴다.**
2. **기능·시나리오에 더미가 있으면 금지.** 더미가 섞이면 *실제로는 에러인데 정상처럼 보여* 결함을 감춘다.
3. 실패·빈 응답은 mock·더미·목업으로 덮지 않고 **실패로 그대로 드러낸다**(README 원칙 1 "폴백 금지"와 같은 취지).

### ⚠️ 실제 사고 (2026-07-15) — 더미가 결함을 가린 사례

시나리오 테스트를 `TEST_SOURCE_IP`·`TEST_SOURCE_PRIVATE_KEY` **미주입**으로 돌렸다.
그러면 `fixtures/test-data.ts` 의 기본값(`ip=''`, `password='e2e-dummy-pass'` — 소스 SSH가 가짜)으로 실행된다.
소스 수집이 가짜라 마이그레이션이 **"행 없음"으로 실패**했는데, 더미가 그 결함을 정상처럼 가렸다.

→ **기능·시나리오는 실서버 전제 필수값이 비면 fixture 기본값으로 진행하지 말고 즉시 실패시킨다.**
`BASE_URL`·`TEST_SOURCE_IP`·`TEST_SOURCE_PRIVATE_KEY` 등이 없으면 더미로 조용히 도는 대신 명시적으로 죽여, 사람이 "이건 실서버로 안 돌았다"를 바로 알게 한다.

## B. 에이전트별 주의사항

### B-1. cm-beetle — 인프라 이름 충돌

cm-beetle 은 타깃 인프라를 **타깃 모델에 적힌 이름(기본 `infra101`, 또는 `infra{n}`)** 으로 만든다.
같은 이름으로 재생성하면 충돌하고, **죽은 이전 인프라와 구분이 안 돼 "거짓 통과"** 가 난다
(노드가 죽은 `infra101` 을 보고 "EC2 생성 성공"으로 판정).

→ **워크플로우 툴에서 인프라 이름을 유니크하게 바꿔 전달**하고, 실행 전 **이전 인프라를 정리**한다.
이 저장소는 이미 그렇게 한다 — `features/scenarios/워크플로우툴-파라미터변경.feature`,
`features/scenarios/마이그레이션전체흐름.feature`, `steps/software.steps.ts`
(기본값 `infra101` 대신 바꾼 이름으로 워크로드가 생겼는지, **이번 실행이 만든 노드 id** 로 확인).

### B-2. cm-cicada — 워크플로우 이름 유일성 + request_body 방식

- **동일 이름 워크플로우 저장 불가** — cm-cicada 는 워크플로우 이름에 UNIQUE 제약이 있다.
  같은 이름으로 다시 저장하면 거부되고, 에러 모달이 남아 뒤 조작을 가린다.
  → `uniqueName()` / `RUN_ID` 접미사로 **유니크 이름**을 쓴다(`support/naming.ts`).
- **request_body 방식** — v0.5.1 에서 task 의 request_body 는 **참조 / 조합 / 리터럴** 폼을 가진다.
  방식별로 값이 어떻게 조립·보존되는지 주의한다(저장 시 `{}` 소실·참조 라운드트립 등).
  (배경: `cmig-workflow` → `notion/cm-bf-ep-전환워크플로우웹플래닝고도화/017_태스크본문전달방식현행화/`)

### B-3. ★ 추천 후보 선정 — 스펙·이미지가 모두 채워진 후보를 골라야 한다 (2026-07-15, 중요)

인프라 마이그레이션은 **스펙을 비롯한 모든 정보가 채워진 타겟**으로 등록해야 정상 동작한다.
그런데 **cm-beetle 추천 목록에는 일부 항목(스펙·이미지)이 비어(placeholder `"string"`) 추천되는 후보가 섞여 있다.**
빈 후보로 타겟 모델을 만들면 **저장은 되지만**(request_body 는 보존됨) beetle 이 **이미지를 못 풀어 인프라 생성이 실패**한다.

**실제 증거 (2026-07-15)** — 저장된 마이그레이션 워크플로우의 인프라 task `request_body` 에서
`targetInfra.name`·`connectionName` 은 채워졌으나 `nodeGroups[0].cspImageName="string"`, `imageId="string"` 로 비어 있었고,
그 결과 **MCI 가 생성되지 않았다**(비용 자원 0).

→ **목록 선정이 중요하다.**
- **E2E**: 추천 결과에서 `cspImageName`/`imageId` 등이 placeholder(`"string"`)·빈 값인 후보를 **거르고 완전 충전 후보를 고르도록 보정**한다.
  고를 후보가 없으면 **실패로 드러낸다**(더미로 채우지 않는다 — A 규율과 동일).
- **웹 사용자**: 빈(placeholder) 후보를 피하고 스펙·이미지가 채워진 후보를 고른다.

> "빈 결과를 오류로 착각"하는 문제와는 다르다. 여기서는 **후보가 채워진 것처럼 보이지만 스펙·이미지가 placeholder** 인 것이 핵심이다.

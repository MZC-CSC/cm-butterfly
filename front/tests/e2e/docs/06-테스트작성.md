# 06. 테스트 작성

---

## 3계층 — "로그인한다"가 어느 화면인지 어떻게 아는가

시나리오 문장에는 화면 위치가 없다. 위치·요소는 아래 2·3계층에만 있다. **이 분리가 유지보수성의 전부다.**

```
① .feature (한국어)          ← 사람이 읽고 쓰는 "의도"
     먼저 "cmiguser"로 로그인한다
              │  (문장 패턴으로 매칭)
              ▼
② steps/*.ts                  ← 문장 ↔ 액션 연결
     Given('{string}로 로그인한다', async ({ page }, id) => {
       const login = new LoginPage(page);
       await login.goto();
       await login.login(id, pw);
     });
              │
              ▼
③ pages/*.ts (Page Object)    ← "어디(URL)/어떻게(셀렉터)". 화면이 바뀌면 여기만 고친다
     class LoginPage {
       static path = '/auth/login';
       idInput = getByTestId('login-id');
     }
```

**화면이 바뀌면 대개 ③만 고친다. 한국어 시나리오(①)는 그대로 둔다.**

---

## 새 기능을 추가할 때

1. `features/{도메인}/{기능}.feature` 에 **한국어로 동작을 정의**한다(태그 포함).
2. 새 문장이 필요하면 `steps/{도메인}.steps.ts` 에 스텝을 추가한다(기존 스텝 최대 재사용).
3. 새 화면·요소면 `pages/{도메인}.page.ts` 에 URL·셀렉터를 추가하고,
   **화면 소스(.vue)에 `data-testid` 를 부여한다.**
4. `npx bddgen && npx playwright test --grep @{도메인}` 으로 확인.

## ★ 지켜야 할 규칙

### 1. 폴백 금지 — 실패는 실패로 드러낸다

`@unit`·`@scenario` 는 **mock·더미로 덮지 않는다.** 실제 결과를 그대로 검증한다.

폴백을 넣으면 **깨졌는데도 정상처럼 보인다.** 예전에 소프트웨어 추천이 실패하면 더미 추천 데이터로 대체해서,
cm-grasshopper 가 죽어도 화면에는 그럴듯한 추천이 떴다. 그건 버그를 감추는 짓이다.

### 2. 안정성은 testid 로 확보한다 (문구가 아니라)

```ts
// ✗ 나쁨 — 화면 문구가 바뀌면 깨지고, 여러 요소에 걸려 strict mode 위반이 난다
m.locator('...').or(m.getByPlaceholder('Time'))     // "Time" 과 "Test Run Time" 둘 다 걸림

// ✓ 좋음 — testid 전용
m.locator('input[data-testid="load-config-rampup-time"]')
```

**testid 가 없으면 화면 소스에 부여한다.** 이 저장소는 그렇게 하기로 했다.

> ⚠️ 단, `p-button-modal` 같은 **래퍼 컴포넌트에 붙인 testid 는 "보이는 요소"로 잡히지 않는다.**
> 모달의 열림/닫힘은 **안에 있는 실제 입력칸**으로 판정한다.

### 3. 거짓 통과를 경계한다

**"행이 보인다" ≠ "이번 실행이 만들었다".**

- 이름이 겹치면 *이전 런이 남긴 것*이 단언을 만족시킨다 → 생성 이름에 `uniqueName()` 을 쓰고, **확인도 같은 이름으로** 한다.
- 상태까지 본다 → 워크로드는 행 존재가 아니라 **`Running`** 인지 확인한다.
- 늘 떠 있는 요소로 검증하지 않는다 → 실행 이력은 *워크플로우 목록* 표가 아니라 **이력의 State 셀**로 본다.

### 4. 목록은 페이지네이션을 넘겨 가며 찾는다

15행씩 끊기고 검색창은 필터로 동작하지 않는다.
`TablePagination.expectRowSomewhere()` 를 쓰면 페이지를 넘겨 찾고 **몇 페이지에서 찾았는지도 남긴다**.

### 5. 요금 보호

- 추천은 **small급 이하**로 강제한다(`TEST_TARGET_MAX_CLASS`).
- 실제 자원을 만들거나 지우는 테스트는 **`@costly`** 를 붙인다(기본 제외).
- 마이그레이션 워크플로우는 **재시도 금지** — 누를 때마다 EC2가 늘어난다.
- 시나리오는 끝에 **반드시 정리**한다.

### 6. 정리(teardown)는 진짜로 지워야 한다

예전 정리 코드는 Page Object 를 `as unknown as Record<...>` 로 캐스팅한 뒤
`selectInfra?.()` 처럼 **존재하지 않는 메서드**를 옵셔널 호출해서, 아무것도 지우지 않고 조용히 통과했다.

타입을 그대로 두고 실제 메서드를 부른다. **컴파일러가 오타를 잡게 하는 게 요점이다.**

## 이름 짓기

honeybee·cicada 등은 이름에 UNIQUE 제약이 있다. 실행마다 접미사를 붙인다.

```ts
import { uniqueName } from '../support/naming';

await source.createSourceGroup(uniqueName('e2e-src-group'));   // e2e-src-group-842461
// 확인도 반드시 같은 이름으로
await source.expectGroupListed(uniqueName('e2e-src-group'));
```

## 스텝 사이에 값 넘기기

파일이 달라도 같은 값을 봐야 하면 `support/world.ts` 의 `scenarioState` 에 둔다.
(모듈 지역변수에 두면 다른 스텝 파일에서 안 보인다.)

```ts
scenarioState.sourceModelName = uniqueName(name);   // 소스모델 저장 스텝
...
await models.selectModel(scenarioState.sourceModelName);  // 추천 스텝
```

## 디렉토리

```
front/tests/e2e/
├─ docs/               ← (이 문서들)
├─ README.md           ← 설계 정본 + 빠른 사용법
├─ playwright.config.ts
├─ features/           ← ★ 한국어 Gherkin (사람이 정의)
│   ├─ _seed/          ← 데이터 준비
│   ├─ {도메인}/        ← 기능 테스트 @unit
│   └─ scenarios/      ← 전체 흐름 @scenario
├─ steps/              ← 문장 → 액션
├─ pages/              ← URL · 셀렉터 (화면 변경의 충격을 여기 가둔다)
├─ fixtures/test-data.ts   ← 계정·스펙·환경변수
└─ support/
    ├─ pagination.ts   ← 페이지 넘겨 행 찾기
    ├─ naming.ts       ← uniqueName / RUN_ID
    ├─ world.ts        ← 스텝 간 공유 상태
    ├─ fixtures.ts     ← 증거 캡처 · 요청 기록
    └─ globalSetup.ts  ← RUN_ID 고정
```
